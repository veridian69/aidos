# Migration v1.4.0 → v2.0.0

## Summary

v2.0.0 is a major revision: the WHAT/HOW split made structural. Solutions become
strictly functional (new S10 — the system under design is one undivided black box;
no internal component may be named). Titles must state problems, not parts (new
Core C14; Breakdown B6). Problems gain purpose grounding (P12) and Epic goal
altitude (P13). The Tech Design section becomes optional *direction* at
Feature/Story scale, with the architect's discretion recorded.

**This migration is best-effort and offered, not forced.** An artifact that stays
on v1.x keeps its `AIDOS Version` stamp and remains valid under the version it was
built with — the v2 Auditor will not assess v1.x artifacts against v2 rubrics (it
recommends migration and stops). Migrate artifacts as you reopen them for active
work. Steps 2, 3, and 5 are content-level rewrites: apply them with the human in the loop,
presenting diffs before writing (per the Builder's per-file upgrade flow).

## File renames

None automatic. See Content changes step 5 — title-driven renames are proposed to
the human, never executed unilaterally.

## Content changes

Apply to one artifact file at a time, in this order:

1. **All files — no structural section changes.** v2.0.0 adds no sections and
   removes none. A combined `feature.md` (or Story file) that already contains a
   Tech Design section keeps it — existing inclusions are grandfathered as
   implicit architect judgment; do NOT add retroactive omission lines to files
   that have content in the slot. If a Tech Design slot exists but is empty or
   placeholder-only, ask the human for the discretion decision — fill in
   direction, or record the one-line omission with their reasoning (never invent
   it).

2. **Problem artifacts.** Step 2a applies at every scale (at Feature/Story, a
   reference to the Epic's purpose satisfies P12); step 2b applies at Epic scope
   only.
   a. P12 check: if the Problem Statement opens with implementation pain
      ("X has structural issues, replace with Y") rather than who the system
      serves and what success looks like, draft a purpose-grounding opening from
      content elsewhere in the artifact (Stakeholders, Impact). If the purpose is
      not recoverable from the artifact, add an Issue (status OPEN): "P12 — system
      purpose not stated; needs stakeholder input."
   b. P13 check: move implementation-shaped Goals (architecture/codebase outcomes)
      to the Overflow Log, tagged with destination (Feature scope or Tech Design)
      and status PARKED. Moved, not deleted. Renumber remaining Goals; update any
      references to the old numbering elsewhere in the artifact. Cross-artifact
      references to moved Goals are caught by the next coherence audit of the
      downstream artifact.

3. **Solution artifacts (and the Solution section of combined files).**
   Apply S10: for every sentence naming an internal component, module, service,
   layer, or technical role of the system being built (active or passive voice):
   - rewrite it as the observable outcome it produces, OR
   - move it to the Tech Design (Epic: the separate document; Feature/Story: the
     Tech Design / Technical Approach section), preserving the content.
   The component vocabulary moves with the content — after migration the Solution
   names no internal parts. Note: the S10 slot previously held "Implementation
   neutrality" (retired v1.3.0, subsumed by C13); old and new S10 are different
   criteria — no v1.x artifact carries old-S10 content, so no reverse migration
   applies.

4. **Tech Design artifacts:** no content changes. (The "direction" reframe changes
   how new Tech Designs are scoped, not what existing ones contain.)

5. **Titles (optional — most invasive).** Where a Feature folder or Story file is
   component-named (`f3-resolver/`, `f1s2-config-writer.md`), propose a
   problem-statement retitle and the corresponding file/folder rename to the
   human, including the parent-link updates in affected files. Wait for explicit
   confirmation before renaming — links may exist outside the repository
   (Confluence, tickets). If declined, record an Issue: "C14/B6 — title below
   altitude; retitle deferred."

## Metadata changes

  (per the Builder's per-file upgrade flow).
